What is GIT? Why do we need source code Version Control System?

GIT is a Distributed Version Control System (DVCS) which allows clients to mirror a project's file repository, including its version history. This allows project participants to collaborate on the deliverables of a project, while enabling administrators to control access and modification rights to project files. Additionally, unlike Local Version Control Systems, which exist exclusively on the client system, or Centralized Version Control Systems, which are a single point of failure when they go down, Distributed Version Control Systems provide redundancies and backups of the project files (via the mirrored file repositories on each client system) so that a failure of any given system does not prevent clients from continuing to work and the project's files are protected against data loss.1All of the above is why we need to use a Version Control System (VCS). It provides backups, tracks changes, allows users and administrators to return to earlier versions of project releases or individual project files, and allows collaboration as multiple people can work on the same files while merging changes back into the remote repository.2

Describe following terms: 1) local repo 2) remote repo 3) how to use GIT in Team Project - multiple developers work on the same project at the same time.

Remote repositories are the files of a project, along with version histories, stored on the network or the internet. Data files can be checked out of the remote repository and pulled by a client into a local repository, where a mirror image of the remote repository is maintained, thus allowing a project member to work on project files whether or not network access is maintained. When a client is ready to update a new version of the file, the local repository files are checked in and merged into the remote repository.3 When multiple developers are working on a project in GIT, the current project files are kept in a Master branch, with project developers working off of development branches which they clone to their local repositories. When a developer seeks to make a modification to a file, it is checked out in the remote repository and discrete features are modified in a local branch. Upon completion, the changes are pushed or merged back into the remote repository. When a project release is ready, a new release branch is created with the project files in the DEV branch. When the production release has been certified, the release branch is merged into the master branch.4
